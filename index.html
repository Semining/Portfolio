POINT 3가지
Point 01

What I Learned
[ 화면을 설계하기 전에, '누가 무엇을 볼 수 없는가'를 먼저 물어야 한다 ]
Figma를 열기 전에 AS-IS 프로세스를 분석하면서 Amazon 보안팀이 제3자 벤더의 이상 행위를 실시간으로 볼 수 없다는 구조적 빈틈을 먼저 파악했습니다. 화면을 설계한다는 것은 UI를 그리는 일이 아니라, 누가 어떤 정보를 언제 봐야 하는지를 결정하는 일이라는 것을 이 프로젝트에서 처음 체감했습니다.
[ 요구사항은 수치로 끝맺어야 팀 내 합의가 된다 ]
"모니터링을 강화하자"는 방향만 있었을 때는 팀 내 논의가 계속 겉돌았습니다. "이상 탐지 후 30초 내 보안팀 알림, 발견사항 30일 이내 조치"처럼 수치로 기준을 세우자 요구사항 문서가 실제 검증 가능한 산출물이 되었습니다. 기능 요건에 수치가 없으면 개발자도 완료 기준이 없다는 것을 배웠습니다.
[ 비용 분석은 TO-BE 설계의 우선순위를 정하는 도구다 ]
AS-IS 연간 416시간·$88,400이라는 숫자는 단순 비교용이 아니었습니다. 수동 취약점 스캔·위협 분석·벤더 알림 각 항목이 얼마나 소요되는지 분해한 뒤에야 "무엇을 먼저 자동화할 것인가"를 결정할 수 있었습니다. 비용 데이터가 TO-BE의 기능 우선순위와 직접 연결되어야 한다는 것을 배웠습니다.

Ac.com 숙박 예약 웹사이트

비즈니스 로직을
시스템으로 정의하다

숙박 예약 플랫폼 Ac.com에서 주문(Order) 도메인 전체를 담당했습니다. "날짜별 만실을 어떻게 처리할까", "취소된 예약은 어떤 상태로 남겨야 할까" — 비즈니스 질문을 직접 정의하고, 그 답을 시스템 구조로 설계했습니다.

GitHub — rynne4976-art/Accommodation
POINT 3가지
Point 01
요구사항을 데이터 구조로 번역하는 능력
"날짜별 만실을 달력에서 보여줘"라는 비즈니스 요건이 OrderStayDate 테이블 설계로 이어지는 과정을 직접 경험했습니다. 개발팀에게 요구사항을 구조적으로 전달할 수 있습니다.
Point 02
엣지 케이스를 사전에 정의하는 능력
취소된 예약은 어떤 상태로 남겨야 할까? — 비즈니스 질문에 답하면서 시스템 설계를 구체화했습니다. 예상치 못한 상황을 미리 고려하는 것이 중요합니다.
Point 03
도메인 간 의존성과 데이터 흐름 이해
Order가 Accom·Member·Review와 어떻게 연결되는지 알기 때문에, 팀 간 인터페이스 정의와 의존성 이슈를 먼저 발견하고 조율할 수 있었습니다.
비즈니스 요건 → 시스템 설계 — 번역의 과정
비즈니스 질문
"날짜가 꽉 찼는지 달력에서 바로 보여줘"
예약 폼 진입 시 이미 만실인 날짜를 달력에서 회색으로 막아줘야 한다. 어떻게 판단할까?
→
시스템 설계 결정
날짜 범위가 아닌 날짜별 레코드 저장
체크인~체크아웃을 범위로만 저장하면 날짜별 중복 판단이 복잡해진다. 하루씩 OrderStayDate 레코드로 저장하면 GROUP BY 집계로 만실 여부를 단순하게 조회할 수 있다.
→ OrderStayDate 테이블 설계
비즈니스 질문
"예약을 취소하면 그 날짜는 다시 열려야 해"
취소된 예약의 날짜가 만실 카운트에 포함되면 안 된다. 하지만 예약 기록 자체는 남아야 한다.
→
시스템 설계 결정
삭제가 아닌 상태 변경 + 쿼리 조건 제외
OrderStayDate를 삭제하지 않고 Order의 상태를 CANCEL로 변경한다. 만실 조회 쿼리에서 CANCEL 상태를 제외하는 조건을 추가해 데이터 일관성과 이력 보존을 동시에 달성한다.
→ orderStatus <> CANCEL 조건
비즈니스 질문
"날짜를 바꾸면 기존 예약 점유는 어떻게 되나?"
체크인·체크아웃을 수정하면 기존 날짜 점유 레코드가 남은 채 새 날짜 레코드가 추가되면 안 된다.
→
시스템 설계 결정
수정 시 StayDate 전체 초기화 후 재생성
수정 요청이 들어오면 기존 OrderStayDate를 먼저 모두 제거하고(clearStayDates), 새 날짜 범위로 재생성(syncStayDates)한다. 검증도 수정 대상을 제외한 상태로 재실행한다.
→ clearStayDates() + excludeOrderItemId
비즈니스 질문
"관리자가 주문을 검색하고 상태를 바꿀 수 있어야 해"
주문 ID·이메일·회원명으로 검색하고 ORDER/CANCEL 상태를 변경할 수 있어야 한다. 단 취소된 예약을 다시 ORDER로 되돌리는 건 불가능해야 한다.
→
시스템 설계 결정
동적 검색 쿼리 + 상태 전이 규칙 코드화
QueryDSL로 검색 조건을 동적으로 조합했다. 관리자 상태 변경 시 "CANCEL → ORDER 전이 불가" 규칙을 서비스 레이어에서 명시적으로 차단해 운영 정책을 코드로 강제했다.
→ updateOrderStatusByAdmin() 정책 로직
사전에 정의한 엣지 케이스 7가지 — 테스트 시나리오의 원형
예약 불가 상태의 숙소에 예약이 들어오면?
reserveStatCd가 "Y"가 아닌 숙소는 예약 생성 즉시 차단. 운영 중단 숙소에 예약이 쌓이는 상황을 방지한다.
결정 — 가장 첫 번째 검증으로 배치, 이후 로직 실행 안 함
체크아웃이 체크인보다 앞선 날짜로 들어오면?
프론트 UI에서 막더라도 API 직접 호출 가능성이 있다. 서버에서도 반드시 날짜 순서를 검증한다.
결정 — 서버 사이드 검증 필수, 프론트만 믿지 않음
최대 수용 인원보다 많은 인원으로 예약하면?
숙소마다 maxGuestCount가 다르다. 초과 시 몇 명까지 가능한지 메시지에 포함해 UX를 개선한다.
결정 — 에러 메시지에 최대 인원 수 동적으로 포함
오늘 날짜인데 이미 체크인 시간이 지났으면?
날짜가 오늘이더라도 체크인 시간이 지나면 예약 불가. 숙소마다 체크인 시간이 다르므로 운영 정책에서 참조한다.
결정 — LocalTime.now() vs policy.getCheckInTime() 비교
운영 기간 밖 날짜를 선택하면?
숙소마다 운영 시작일·종료일이 다르다. 체크인이 운영 시작일 이전이거나 마지막 숙박일이 운영 종료일 이후면 차단한다.
결정 — operationStartDate / operationEndDate 정책 기반 검증
운영하지 않는 요일이 선택 구간에 포함되면?
숙소는 특정 요일만 운영할 수 있다. 선택한 날짜 범위에 비운영일이 끼어 있으면 전체 예약이 불가하다.
결정 — operationDayList Set으로 날짜별 포함 여부 확인
선택한 날짜 중 하루라도 만실이면?
3박 예약에서 2일은 가능하고 1일만 만실이어도 전체 예약 불가. 어느 날짜가 문제인지 메시지에 포함한다.
결정 — 날짜별 countReservedByDate() 루프, 문제 날짜 메시지 반환
이 7가지는 단순한 개발 예외 처리가 아닌 기획 단계에서 정의되는 비즈니스 규칙입니다. 개발 경험을 바탕으로 이런 엣지 케이스를 미리 발견하고, IT 기획자로서 기획서에 명시하거나 QA 테스트 시나리오로 만들어볼 수 있었습니다.

What I Learned
[ 요건의 모호함은 엔티티 수로 돌아온다 ] "숙소마다 운영일이 다르다" 한 문장이 시스템에서 2개의 엔티티로 쪼개졌습니다. 기획 단계에서 경계 조건을 명시하지 않으면 그 해석 비용을 개발자가 설계로 떠안게 된다는 것을 배웠습니다.
[ 비즈니스 규칙은 어느 레이어에서 보장할지 기획이 결정해야 한다 ] "리뷰는 1인 1회"라는 규칙을 앱 코드에서만 막을지, DB 제약까지 걸지는 기술 결정처럼 보이지만 실은 규칙의 중요도를 기획자가 먼저 정의해야 결정됩니다. 규칙의 강도를 문서화하는 것이 기획자의 몫임을 배웠습니다.
[ 상태 전이를 정의하지 않으면 예외 케이스는 개발 중에 발견된다 ] 주문 취소 후 재취소, 이미 마감된 날짜 예약 시도 등은 개발 중 뒤늦게 발견되기 쉬운 케이스입니다. 상태 머신을 기획 산출물로 먼저 그려두면 엣지 케이스가 테스트 시나리오의 목차가 되고, 개발 일정 리스크도 줄어든다는 것을 배웠습니다.
[ 성능 결정도 기획 요건에서 시작된다 ] 평균 평점을 실시간 계산할지 저장해둘지는 "평점이 얼마나 자주, 얼마나 빠르게 보여야 하는가"라는 기획 요건이 먼저 있어야 결정됩니다. 비기능 요건(응답속도·정합성 허용 범위)을 기획 단계에서 명시하지 않으면 개발자가 임의로 선택하게 된다는 것을 배웠습니다.
[ 개발을 직접 해보면 기획 문서에서 빠진 것이 보인다 ] 이전 프로젝트들과 달리 케이스들을 정의했기 때문에 순조롭게 진행이 가능했습니다. 개발 경험은 기획자에게 "어디서 구멍이 생기는지"를 사전에 감지하는 감각을 줍니다. 요건 정의서를 쓸 때 개발자 관점의 질문을 먼저 던질 수 있게 된 것이 가장 큰 수확이었습니다.
